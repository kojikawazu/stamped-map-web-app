# 開発設計実装方針（MVP開発レベル）

> 本文書はこのプロジェクト（Stamped Map Web App）における**開発方針のベースライン**である。
> スタック：Nuxt.js 3 + TypeScript + Tailwind CSS v4 / Supabase (PostgreSQL + Auth) / Vercel

## 全体

- それぞれの言語仕様のコーディング規約に準ずる。
- リンター・フォーマッターを導入し、コード品質を自動で担保する。
- 環境変数で環境差分を管理する（.env）。

---

## 開発フロー方針

### ワークフロー

1. **設計** → レビュー待ち（レビュアーの承認を得るまで実装に進まない）
2. **実装** → レビュー待ち（レビュアーの承認を得るまでプッシュしない）
3. **プッシュ・PR作成** — 実装レビュー完了後に実施
4. **PR承認・マージ** — 意思決定者が行う

### 権限（承認ゲート）

| フェーズ | レビュー前にできること | レビュー承認後にできること |
|---|---|---|
| 設計 | ドキュメント作成 | 実装着手 |
| 実装 | コーディング・ローカルコミット | プッシュ・PR作成 |
| PR | — | マージ（意思決定者のみ） |

### 意思決定者の指示

- 意思決定者からの変更指示には従うこと。
- 設計・実装の方針に疑問がある場合は確認を取るが、最終判断は意思決定者に委ねる。

## Git運用方針

### ブランチ戦略

- **GitHub Flow** を基本とする。
- `main` ブランチへの直接コミットは禁止。必ずブランチを切る。
- ブランチ名はGitHubの命名規約に準ずる（例: `feature/add-user-api`, `fix/login-error`）。

### ワークフロー

1. `main` から作業ブランチを切る
2. 作業ブランチでコミット・プッシュ
3. PR（Pull Request）を作成する — **ここまでは自動化OK**
4. PR承認・マージは **必ず人が行う**（自動マージ禁止）

### リポジトリ構成

- **モノレポ（ルート管理）** を基本とする。

```
/
├── docs/           # ドキュメント（設計書・仕様書など）
├── front/          # フロントエンド（Nuxt.js 3）※ Server Routes含む
├── README.md       # プロジェクト概要（必須）
├── CLAUDE.md       # AI開発ガイド（必須）
└── .gitignore      # Git除外設定（必須）
```

- **一体型** — `front/` のみ。API は Nuxt Server Routes（Vercel Serverless）で完結。

### 必須ファイル

- **README.md** — プロジェクト概要・セットアップ手順
- **CLAUDE.md** — AI開発時のコンテキスト・ルール
- **.gitignore** — 以下を必ず除外する
  - `.env` / `.env.*` — 環境変数・シークレット
  - `node_modules/` — 依存パッケージ
  - `.nuxt/` — ビルド成果物
  - `*.pem` / `*.key` — 秘密鍵・証明書
  - クレデンシャル系ファイル（`credentials.json`, `serviceAccountKey.json` 等）

### Pushしてはいけないもの

- シークレット情報（APIキー・DB接続文字列・トークン等）
- 秘密鍵・証明書ファイル
- ローカル環境固有の設定ファイル
- 大容量バイナリファイル

## ドキュメント方針

### 原則

- ドキュメントは**常に最新の状態を維持する**。コード変更時はドキュメントも必ず更新する。
- 仕様書とコードの乖離を防ぎ、**製造性（仕様書からコードへの再現性）を担保する**。
  - 仕様書を読めば実装方針が明確になるレベルを目指す。

### 修正忘れ注意

- 以下のファイルはコード変更時に**修正漏れが発生しやすい**ため、特に注意する。
  - `/README.md` — ルート直下
  - `/CLAUDE.md` — ルート直下
  - `/docs/11-tasks.md` — タスク進捗
- PRレビュー時に、これらのファイルが更新されているかをチェック項目とする。

### 追加ドキュメントのルール

以下はプロジェクト立ち上げ時には生成しない。該当する要件・設計が発生した時点で別途作成する。

| ドキュメント | 作成タイミング |
|-------------|--------------|
| 環境変数一覧 | 環境変数が確定した設計フェーズ |
| 外部サービス設定手順 | 利用サービスが確定した設計フェーズ |
| 機能設計書（`docs/design/` 配下） | 個別機能の設計時 |
| テスト設計書（`docs/test-design/` 配下） | テスト設計時 |

### 必須運用

- **GitHub Issue** — タスク・バグ・機能要望は必ずIssueとして起票する。
- **タスクファイル** — `docs/` 配下にタスク管理ファイルを作成し、進捗を更新する。
- **README.md** — セットアップ手順・構成変更があれば都度更新する。
- **CLAUDE.md** — AI開発に関わるルール変更があれば都度更新する。

### ドキュメント配置

```
docs/
├── tasks/          # タスク管理ファイル
├── specs/          # 仕様書
├── design/         # 設計書
├── requirements/   # 要求整理
└── discussions/    # ディスカッションレポート
```

## フロントエンド

### コンポーネント設計方針

プロジェクト規模・ドメイン数に応じて以下のいずれかを選択する。

| パターン | 構成 | 採用基準 |
|---|---|---|
| **アトミックデザイン** | Atoms / Molecules / Organisms / Pages | 小〜中規模・ドメインが少ない・UIパーツの再利用性を重視する場合 |
| **ドメイン別構成** | features/ 配下にドメイン単位で分割 | 中〜大規模・ドメインが多い・チーム分割を意識する場合 |

#### アトミックデザイン（デフォルト）

- **Atoms** — ボタン、入力フィールド、ラベルなどの最小UIパーツ
- **Molecules** — Atomsを組み合わせた小さな機能単位（検索バー、フォームグループなど）
- **Organisms** — Moleculesを組み合わせたセクション単位（ヘッダー、カード一覧など）
- **Pages** — ページ全体の構成（Organismsの配置とデータ取得）

#### ドメイン別構成（例外ルート）

- ドメインが多い場合や機能単位での開発・保守が有利な場合に採用する。
- 共有UIパーツは `components/common/` に配置し、ドメイン固有のUIは `features/<domain>/components/` に閉じる。
- 採用時はプロジェクト初期に意思決定者の承認を得ること。

### サーバー/クライアント分離

- 地図コンポーネント（MapLibre）は `<ClientOnly>` でラップし、SSRを回避する。
- ページコンポーネントはCSRを基本とする（地図SPA全体が認証必須かつCSR）。
- ロジックは **Composables** に切り出す。
  - コンポーネントはUIの描画に専念し、状態管理・データ取得・イベント処理は composables に分離する。
  - 例: `useSpots.ts` にフィルタ・ソート・ページネーションロジックを集約。

### ディレクトリ構成（Nuxt.js 3 / アトミックデザイン）

```
front/
├── pages/              # ルーティング（index.vue / login.vue）
├── layouts/            # レイアウト（default.vue / empty.vue）
├── components/         # アトミックデザイン
│   ├── atoms/
│   ├── molecules/
│   └── organisms/
├── composables/        # Composables（状態管理・データ取得）
├── lib/                # ユーティリティ・バリデーション
├── types/              # 型定義
├── middleware/         # ルートミドルウェア
└── server/
    ├── api/            # Server Routes（API エンドポイント）
    └── utils/          # サーバー側ユーティリティ
```

## API設計方針

### 構成

- **Nuxt Server Routes**（`server/api/`）で完結。追加インフラ不要。
- Vercel Serverless Functions として自動デプロイされる。

### 共通方針

- RESTful設計を基本とする。
- エンドポイントの命名は **リソース指向**（例: `/api/spots`, `/api/spots/:id`）。
- レスポンス形式は JSON で統一する（`{ data: ... }` または `{ error: ... }` 形式）。
- 全エンドポイントで `verifyAuth()` による JWT 検証を必須とする。
- `defineEventHandler` でハンドラーを定義し、ロジックは `server/utils/` に切り出して薄く保つ。

## テスト方針

### テスト分類

- **正常系** — 期待通りの入力に対して、正しい結果が返ることを検証する。
- **準正常系** — 想定内の異常入力（バリデーションエラーなど）に対して、適切にハンドリングされることを検証する。
- **異常系** — 想定外のエラー（DB接続断、外部API障害など）に対して、安全に失敗することを検証する。

### テストの原則

- テストを通すためにテストケースやアサーションを捻じ曲げない。
  - テストが失敗した場合、**実装側を修正する**のが正しい対応である。
  - 期待値を実装の出力に合わせて変更するのは禁止とする。
- テストは仕様の証明であり、実装の後追いではない。

---

## エラーハンドリング方針

- バリデーションは必須とする。
- 軽量DDDのドメインクラス・値オブジェクトで防げるものは、そちらで制約を担保する。
  - 例: 値オブジェクトのコンストラクタで不正値を弾く
- ドメイン層で防げない入力（リクエスト形式・型チェックなど）は、ハンドラー層でバリデーションする。

## DB設計方針

### 命名規約

- テーブル名・カラム名は **snake_case** とする。
- テーブル名は **複数形**（例: `users`, `orders`）。
- カラム名は **単数形**（例: `user_name`, `email`）。

### 共通カラム

- 全テーブルに以下のカラムを必須とする。
  - `id` — 主キー（UUID または AUTO INCREMENT）
  - `created_at` — レコード作成日時
  - `updated_at` — レコード更新日時

### 論理削除 vs 物理削除

- 基本方針はプロジェクト要件に応じて選択する。
  - **論理削除** — `deleted_at` カラムを追加（データ復旧・監査が必要な場合）
  - **物理削除** — レコードを直接削除（シンプルさ優先・MVP向き）

### 正規化

- **第3正規化**を基本とする。
- MVPでは過度な非正規化は行わず、パフォーマンス問題が発生してから検討する。

## CI/CD方針

### デプロイ先の選択

プロジェクト要件に応じて、フロントエンド・バックエンドそれぞれのデプロイ先を選択する。

| 対象 | 選択肢 | 採用基準 |
|---|---|---|
| **フロントエンド** | Vercel | Nuxt.jsとの親和性が高い・プレビューデプロイが必要・手軽にMVPを出したい場合 |
| **フロントエンド** | Cloud Run | GCPに統一したい・Vercelの制約（タイムアウト・リージョン等）を超えたい・SSRの細かい制御が必要な場合 |
| **バックエンド** | Cloud Run | コンテナベースでスケーラブルなAPI基盤が必要な場合（デフォルト） |
| **バックエンド** | なし（一体型） | Nuxt Server Routes内で完結する場合（一体型） |

### CI/CD 構成（Vercel + GitHub Actions）

- Vercel のデフォルトビルド・デプロイを利用する（`nuxt build` → Vercel Serverless）。
- GitHub Actions で CI を実行する。

**CI（Push / PR 時）**
- ユニットテスト（Vitest）
- E2E テスト（Playwright）

**CD（main マージ時）**
- Vercel が自動デプロイ

## セキュリティ設計方針

※ パフォーマンスとセキュリティのバランスを意識し、MVP段階で過剰にならない範囲で対策する。

### 認証・認可

- 認証方式はプロジェクト要件に応じて選択する（JWT / セッション / Supabase Auth 等）。
- APIエンドポイントごとにアクセス制御を設定する。
  - 認証不要（公開API）/ 認証必須 / ロールベースの3段階を基本とする。
- JWTを使用する場合、トークン検証のキャッシュでパフォーマンスとのバランスを取る。

### 通信・アクセス制御

- 全通信は **HTTPS** を必須とする。
- **CORS** は許可するオリジンを明示的に指定する（`*` は本番で使用しない）。
- **レートリミット** を導入する。
  - 正常なユーザー利用を阻害しない範囲で設定する。
  - 認証系エンドポイント（ログイン・サインアップ）は特に厳しく制限する。

### インジェクション対策

- **SQLインジェクション** — ORM または パラメータバインドを必須とする。生SQLの文字列結合は禁止。
- **XSS** — ユーザー入力のサニタイズに加え、CSP（Content Security Policy）ヘッダーを設定する。出力エスケープも併用し、多層防御とする。
- **CSRF** — 認証方式に応じて対策を選択する。
  - **セッション認証（Cookie）** — CSRFトークンによる保護を必須とする。
  - **Bearer Token（JWT）認証** — CookieにトークンをセットしないためCSRF対策は不要。ただしトークンをCookieに保存する場合はCSRF対策を適用する。

### シークレット管理

- シークレット情報（APIキー・DB接続情報等）はコードにハードコードしない。
- ローカル環境では `.env`（`.gitignore` に追加必須）。
- 本番環境では **Secret Manager** 等のマネージドサービスを使用する。
