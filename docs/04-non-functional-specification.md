# Non-Functional Specification (非機能仕様書)

個人利用の Web アプリとして、過剰な設計を避けつつ快適に使える水準を定義する。

## パフォーマンス

### レスポンスタイム目標

| 操作 | 目標 | 備考 |
|------|------|------|
| 地図の初期表示 | 3秒以内 | MapTiler タイル読み込み含む |
| スポット一覧取得 | 1秒以内 | API レスポンス（ページネーション適用後） |
| スポット登録・更新・削除 | 1秒以内 | API レスポンス |
| 地図パン・ズーム | 即時（60fps） | MapLibre WebGL レンダリング |
| 検索・フィルター | 1秒以内 | API レスポンス |

### フロントエンド

- メイン画面は CSR（`'use client'`）で構成（MapLibre + 認証ガードのため）
- 初回表示はローディングシェル → 認証確認 → コンテンツ表示 の順序
- 地図コンポーネントは `dynamic import`（SSR対象外）
- MapTiler タイルはブラウザキャッシュを活用

### データベース

- PostGIS の空間インデックス（GIST）を location 生成カラムに作成（SQL migration で管理）
- ページネーションによりクエリ結果を制限（1ページ20件）
- カテゴリフィルター・検索・ソートは DB 側で処理（クライアントに全件送らない）
- 地図マーカー用データは専用エンドポイント（`/api/spots/markers`）で軽量返却

## スケーラビリティ

### 想定データ量

| 指標 | 想定値 | 備考 |
|------|--------|------|
| スポット数 | 〜数千件 | 個人利用で数年分 |
| カテゴリ数 | 〜数十件 | デフォルト5 + カスタム |
| 同時ユーザー数 | 1人 | 個人利用 |

### スケーリング方針

- MVP 時点ではスケーリング設計は不要
- Vercel Serverless Functions は自動スケール
- Supabase 無料枠の制限内で運用
- データ量が増えた場合：PostGIS インデックスとページネーションで対応

## 可用性

| 項目 | 目標 | 備考 |
|------|------|------|
| 稼働率 | Vercel / Supabase のSLA に依存 | 個人アプリのため独自SLA不要 |
| メンテナンス | 随時（告知不要） | 利用者 = 開発者 |
| バックアップ | Supabase 自動バックアップ | 無料プランは直近7日分 |

## 信頼性

### エラーハンドリング方針

| レイヤー | 方針 |
|----------|------|
| フロントエンド | トースト通知でエラーを表示、画面は壊れない（Error Boundary） |
| API Routes | try-catch で例外をキャッチ、適切なHTTPステータスコードを返却 |
| DB | Prisma のエラーハンドリング、接続エラー時はリトライしない（即エラー返却） |

### トークンリフレッシュ

- Supabase JWT のデフォルト有効期限は1時間
- クライアント側の Supabase SDK が自動リフレッシュを処理
- API が 401 を返した場合、クライアントは以下の順序で対応：
  1. Supabase SDK でセッションリフレッシュを試行
  2. 成功 → 新しいトークンでリクエストをリトライ
  3. 失敗 → ログイン画面にリダイレクト

### データ整合性

- Prisma のトランザクションを必要に応じて使用
- カテゴリ削除時にスポットが紐づいている場合は警告（カスケード削除しない）
- バリデーションはクライアント側（UX）+ サーバー側（セキュリティ）の二重チェック

## ユーザビリティ

### UX 基本方針

| 項目 | 方針 |
|------|------|
| ローディング | データ取得中はスケルトンUI or スピナーを表示 |
| エラー通知 | トースト通知（成功=緑、エラー=赤、数秒で自動消去） |
| 空状態 | スポット未登録時にガイドメッセージを表示 |
| 確認ダイアログ | 削除操作時に確認を要求 |
| フォームバリデーション | リアルタイムバリデーション（入力中にエラー表示） |

### 認証ガードのUXフロー

```
1. メイン画面（/）にアクセス
2. ローディングシェルを表示（スピナー or スケルトン）
3. Supabase Auth SDK で getSession() を実行
4. 認証済み → メインコンテンツを表示
5. 未認証 → /login にクライアント側リダイレクト
```

> サーバー側リダイレクトは行わない。初回表示時に一瞬ローディングが見えるが、個人アプリでは許容範囲。

### 対応ブラウザ

| ブラウザ | 対応 |
|----------|------|
| Chrome（最新） | ○ |
| Firefox（最新） | ○ |
| Safari（最新） | ○ |
| Edge（最新） | ○ |
| モバイルブラウザ | Phase 2 |
| IE | × 非対応 |

> WebGL 必須のため、WebGL 非対応ブラウザでは地図が表示されない。

## 環境変数管理

| 変数 | 用途 | 配置場所 |
|------|------|----------|
| `DATABASE_URL` | Supabase 接続文字列（プーリング） | Vercel env / .env.local |
| `DIRECT_URL` | Supabase 直接接続（Prisma Migrate用） | Vercel env / .env.local |
| `SUPABASE_URL` | Supabase プロジェクトURL（`@nuxtjs/supabase` が読む） | Vercel env / .env.local |
| `SUPABASE_KEY` | Supabase Anon Key（`@nuxtjs/supabase` が読む） | Vercel env / .env.local |
| `NUXT_PUBLIC_MAPTILER_KEY` | MapTiler API Key（クライアントに公開） | Vercel env / .env.local |
| `NUXT_PUBLIC_SITE_URL` | サイトURL（Google OAuth リダイレクト先ベースURL） | Vercel env / .env.local |

> `NUXT_PUBLIC_` プレフィックスの変数はクライアントにも公開される（`nuxt.config.ts` の `runtimeConfig.public` にマップ）。
> `DATABASE_URL` / `DIRECT_URL` はサーバー側のみ。クライアントバンドルに含めない。
> `SUPABASE_URL` / `SUPABASE_KEY` は `@nuxtjs/supabase` モジュールが直接読む変数名のため `NUXT_PUBLIC_` プレフィックスは付けない。
> Google OAuth の `GOOGLE_CLIENT_ID` / `GOOGLE_CLIENT_SECRET` はアプリが直接使用しない（Supabase ダッシュボード側で設定）。アプリの環境変数には不要。

## ログ・監視

| 項目 | 方針 |
|------|------|
| アプリケーションログ | Vercel Functions のログ（自動） |
| エラー監視 | MVP では Vercel ダッシュボードで確認 |
| パフォーマンス監視 | MVP では Vercel Analytics（無料枠） |

> 個人アプリのため、外部監視ツール（Sentry等）は当面不要。必要になったら導入。
